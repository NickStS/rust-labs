1. Які способи передачі даних між потоками ви знаєте? Які переваги та недоліки кожного із них?

Канали (mpsc, crossbeam-channel) — потоки спілкуються через FIFO чергу, передаючи власність даних. Переваги: чітка модель власності, відсутність гонок, зручно для pipeline-архітектури. Недоліки: затримки на серіалізації передачі, складніше для малих обсягів даних. Мютекси з спільною памяттю (Arc<Mutex<T>>) — кілька потоків мають доступ до одних даних, синхронізуються через блокування. Переваги: швидко для невеликих структур, природно для shared state. Недоліки: можливі deadlock, contention при високій конкуренції. Атомарні типи (AtomicUsize, AtomicBool) — для простих скалярних значень. Переваги: lock-free, дуже швидко. Недоліки: тільки для простих типів.

2. Які способи спільного доступу до даних між потоками ви знаєте? Які переваги та недоліки кожного із них?

Arc<T> для read-only доступу — спільне володіння через підрахунок посилань. Переваги: безпечно і швидко для незмінних даних. Недоліки: не дозволяє модифікацію. Arc<Mutex<T>> — взаємне виключення, тільки один потік може писати або читати. Переваги: універсально. Недоліки: блокування, можливість deadlock. Arc<RwLock<T>> — багато читачів або один писач. Переваги: масштабується для read-heavy навантажень. Недоліки: писач може страждати від голодування. Atomic-типи — для одиночних значень без блокування. Переваги: найшвидші. Недоліки: обмежений набір типів і операцій. Channels — передача даних замість спільного доступу, у дусі Go.

3. Що станеться з Mutex, якщо потік, який ним володіє, запанікує?

Mutex стає "отруєним" (poisoned). Наступний потік що спробує lock() отримає Err(PoisonError). Це механізм безпеки: панікуючий потік міг залишити дані у некоректному стані, тому Rust змушує інші потоки явно вирішити що з цим робити. PoisonError містить guard, тому при бажанні можна продовжити роботу з даними попри отруєння викликом into_inner(), але це треба робити свідомо.

4. Чи може тип бути Send, але не Sync? Наведіть приклад.

Так. Send означає що значення можна передати у інший потік (передача власності). Sync означає що &T можна передавати між потоками. Класичний приклад — Cell<T> та RefCell<T>: вони Send (можна перенести в інший потік), але не Sync (бо дозволяють внутрішню мутабельність без синхронізації — два потоки з посиланнями на один RefCell могли б одночасно його змінювати). Також mpsc::Receiver<T> є Send але не Sync — можна передати володіння одному потоку, але не дозволити двом потокам читати з нього одночасно.

5. Чому в Rust неможливі класичні "data races", але все ще можливі "race conditions" (логічні помилки послідовності)?

Data race — це коли два потоки одночасно звертаються до памяті і хоча б один пише, без синхронізації. Rust на рівні системи типів забороняє таке через правила запозичення: або багато &T, або одне &mut T. У багатопотоковому контексті це доповнюється трейтами Send/Sync. Race condition — це ширше поняття: коли результат залежить від порядку виконання потоків, навіть якщо доступ синхронізовано. Наприклад, два потоки читають значення лічильника, потім обидва записують +1 — лічильник збільшиться лише на 1 замість 2. Кожен окремий доступ під мютексом коректний, але логіка послідовності неправильна. Rust не може це виявити бо це питання алгоритму, а не безпеки памяті.
