1. Що таке асинхронне виконання?

Асинхронне виконання — це модель де задачі можуть призупинятися у точках очікування (наприклад чекаючи відповідь від мережі або завершення IO операції) і повертати керування рантайму, який може запустити іншу задачу на тому ж потоці. Коли очікувана подія настає, задача продовжує виконання з місця призупинення. Це дозволяє одному потоку обробляти тисячі IO-bound операцій одночасно без створення тисяч системних потоків. У Rust асинхронність реалізована через async/await синтаксис, трейт Future та зовнішні рантайми (tokio, async-std).

2. Чому погано блокувати потік під час асинхронного виконання?

Асинхронні рантайми зазвичай мають обмежену кількість worker-потоків (часто за кількістю ядер CPU). Коли задача блокує потік синхронним викликом (std::fs::read, std::thread::sleep, тяжкі обчислення без await), вона займає весь worker і не дає рантайму запускати інші задачі на ньому. Якщо всі worker-потоки заблоковані, увесь рантайм фактично зупиняється і інші задачі не можуть прогресувати. Для блокувальних операцій треба використовувати tokio::task::spawn_blocking або асинхронні аналоги (tokio::fs, tokio::time::sleep).

3. Для чого існує трейт Future?

Future представляє значення яке буде доступне у майбутньому. Це абстракція над відкладеним обчисленням з єдиним методом poll, що повертає Poll::Pending якщо результат ще не готовий, або Poll::Ready(value) коли готовий. Усі async fn компілюються у структури що реалізують Future. Трейт дозволяє рантайму уніфіковано працювати з будь-якими асинхронними операціями: викликати poll, реєструвати waker для пробудження коли дані будуть готові, комбінувати future-и через комбінатори (join, select, then). Без Future неможливо було б побудувати zero-cost абстракцію над асинхронним IO у Rust.

4. Чому існує Pin та Unpin в Rust?

Pin гарантує що значення не буде переміщене у памяті після закріплення. Це критично для self-referential структур — структур що містять посилання на свої власні поля. Async fn компілюється у state machine яка може містити такі самопосилання (наприклад змінна що живе через .await і одночасно є в локальному стеку). Якщо таку структуру перемістити у памяті, посилання стане недійсним і виникне UB. Pin<&mut T> забороняє переміщення через safe API.

Unpin — це маркер-трейт що означає "переміщення цього типу безпечне навіть з-під Pin". Більшість типів автоматично Unpin (числа, рядки, Vec тощо). Future створені async fn зазвичай не є Unpin тому що state machine може містити самопосилання. Pin потрібен переважно при роботі з future-ами на низькому рівні; коли пишеш звичайний async/await код про нього навіть не доводиться думати.
