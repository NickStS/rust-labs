1. Для чого Send трейт?

Send позначає що значення типу можна безпечно передати з одного потоку в інший — переміщенням володіння. Якщо тип реалізує Send, компілятор дозволить передати його у замикання що виконується в іншому потоці (наприклад, у std::thread::spawn). Це фундаментальна гарантія thread-safety на рівні системи типів: код що порушує безпеку передачі між потоками просто не скомпілюється.

2. Для чого Sync трейт?

Sync позначає що тип безпечно використовувати з кількох потоків через спільне посилання &T. Іншими словами, &T є Send тоді й тільки тоді коли T є Sync. Це потрібно для типів які кілька потоків можуть одночасно читати або викликати їх методи через спільне посилання — наприклад Mutex, RwLock, Atomic-типи або просто незмінні структури.

3. Яка між ними різниця?

Send стосується передачі володіння (move) між потоками. Sync стосується одночасного спільного доступу через посилання. Тип може бути Send без Sync (наприклад Cell — переносити можна, але одночасно з двох потоків модифікувати через спільне посилання — ні). Теоретично тип може бути Sync без Send хоча на практиці такі випадки рідкісні (наприклад MutexGuard на деяких платформах). Mutex<T> є Sync навіть якщо T тільки Send, тому що сам Mutex синхронізує доступ.

4. Чому вони існують?

Вони існують щоб винести гарантії thread-safety на рівень системи типів і перевіряти їх під час компіляції а не у рантаймі. Це дозволяє Rust обіцяти "fearless concurrency": якщо код компілюється то у ньому немає data races. Це auto-traits — компілятор автоматично визначає чи реалізує тип Send/Sync на основі його полів, тому розробникам зазвичай не треба робити нічого додатково.

5. Чому деякі типи їх не імплементують?

Тому що їх внутрішня реалізація небезпечна для відповідного сценарію. Rc використовує неатомарний лічильник посилань тому не Send і не Sync. Cell і RefCell дозволяють модифікацію через спільне посилання без синхронізації тому не Sync. Сирі вказівники не Send і не Sync бо компілятор не може довести безпеку. MutexGuard не Send бо деякі ОС вимагають звільняти блокування тим самим потоком який його взяв. Заборона Send/Sync для цих типів — це гарантія що їх неможливо випадково використати неправильно.
