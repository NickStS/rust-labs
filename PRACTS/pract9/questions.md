1. Для чого використовується крейт futures?

futures — це крейт що містить базові абстракції та утиліти для роботи з асинхронним кодом які не входять у std. Серед основних: трейти Stream, Sink, Future-комбінатори (then, map, and_then), методи для роботи з потоками задач (buffer_unordered, for_each_concurrent, try_for_each_concurrent, select_all), синхронізаційні примітиви для async (oneshot, mpsc), а також зручні комбінатори try_join, select, join. Крейт існує бо std має лише сам трейт Future без екосистеми навколо. futures доповнює std і не привязаний до конкретного рантайму, тому код з ним переноситься між tokio, async-std та іншими.

2. Які інструменти для полегшення роботи із асинхронним кодом є у tokio?

tokio::fs та tokio::io — асинхронні аналоги стандартного IO з трейтами AsyncRead/AsyncWrite. tokio::net — TCP, UDP, Unix sockets. tokio::sync — Mutex, RwLock, Semaphore, Notify, mpsc/broadcast/watch/oneshot канали саме для async-контексту. tokio::time — Sleep, Interval, timeout, які знають про waker. tokio::task — spawn, spawn_blocking, JoinSet, JoinHandle, LocalSet. tokio::select! — макрос для очікування першого з кількох future. tokio::pin! — макрос для зручного пінінгу на стеці. tokio::main та tokio::test — атрибути для запуску async функцій. tokio_util — додаткові утиліти типу StreamReader, FramedRead, CancellationToken.

3. Чому погано використовувати синхронний Mutex між await викликами?

Синхронний Mutex (std::sync::Mutex) блокує OS-потік. Якщо взяти його, потім зробити .await і задача буде призупинена, синхронний guard залишиться захопленим. Worker-потік на якому виконувалася задача звільниться, але блокування все ще тримається. Тепер інший потік виконавця може запустити іншу задачу яка спробує взяти той самий Mutex — і заблокується. Якщо весь worker pool заблокується на одному Mutex поки задача-власник чекає на await, виникне deadlock. Для таких випадків треба використовувати tokio::sync::Mutex який передбачений для async і вміє правильно звільняти потік під час очікування.

4. Чому аргументи асинхронної задачі, яку ми "спавнимо", мають бути 'static?

tokio::spawn передає Future у виконавець який може запустити її на будь-якому worker-потоці і виконувати її будь-який час — навіть після того як функція що спавнила вже завершилася. Якщо аргументи містять не-'static посилання, ці посилання можуть стати дійсними після того як їх володілець був звільнений. Rust borrow checker не може перевірити час життя посилання що передається через runtime у іншому потоці, тому єдина гарантія яку він приймає — це 'static, тобто посилання живе протягом усієї програми. Для передачі даних у спавнену задачу їх клонують (Arc::clone, String::clone) або переміщають через move closure.
