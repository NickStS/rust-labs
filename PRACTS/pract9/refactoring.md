Що було покращено у порівнянні з pract7

1. Читання stdin — тепер асинхронне
Було: std::io::stdin().lock().lines() — блокувальний синхронний виклик у async-контексті, який міг заблокувати worker-потік tokio. Стало: tokio::io::stdin() з BufReader і AsyncBufReadExt::lines() — повністю асинхронне читання, не блокує рантайм.

2. Читання файлу зі списком URL — асинхронне
Було: std::fs::read_to_string у синхронному режимі. Стало: tokio::fs::File::open + AsyncBufReadExt::lines() — стримінг рядок-за-рядком без завантаження всього файлу в памʼять одразу.

3. Завантаження тіла відповіді — стримінг замість read_to_end
Було: resp.bytes().await — завантажує всю відповідь у памʼять перед записом. Для великих файлів це означає тримати весь файл у RAM. Стало: resp.bytes_stream() + futures::TryStreamExt::try_next + потокова запис у файл. Памʼять обмежена розміром одного chunk незалежно від розміру файлу.

4. Заміна buffer_unordered + Semaphore на for_each_concurrent
Було: stream::iter(...).buffer_unordered(N) додатково з Semaphore — дублювання обмежень, заплутаніший код. Стало: futures::stream::for_each_concurrent(N, ...) — один параметр контролює конкурентність, ідіоматичніше для випадку коли результат задач не потрібен.

5. Створення вихідної директорії — асинхронне
Було: std::fs::create_dir_all. Стало: tokio::fs::create_dir_all.

6. Reqwest pool tuning
Додано pool_max_idle_per_host = concurrency щоб переиспользувати TCP-зʼєднання при множинних запитах до одного хоста.

7. Семафор передається через Arc::clone у замикання
Це уникає копіювання Permit і дозволяє завантажити permit тільки коли задача почала виконуватися всередині concurrent stream.

Що НЕ було потрібно змінювати

mio — це низькорівнева бібліотека для роботи з event loop ОС (epoll, kqueue, IOCP). tokio вже використовує її під капотом для своїх async-сокетів. Вручну її використовувати потрібно лише якщо пишеш власний рантайм або потрібен дуже специфічний контроль над event loop. Для нашого CLI завантажувача абстракцій tokio + reqwest достатньо.

Не використовувалися Stream-комбінатори типу map+collect для збору результатів, бо результатом задачі є тільки запис у файл і логування — нічого збирати не потрібно. for_each_concurrent якраз для таких випадків.

Не використовувалися tokio::sync::Mutex чи RwLock — спільний стан між задачами відсутній (semaphore це достатньо для контролю concurrency, а інші дані передаються через clone).
