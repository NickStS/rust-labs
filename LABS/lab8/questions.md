1. Які параметри асинхронного рантайму найбільш впливають на час виконання програми? Обгрунтуйте свою відповідь.

worker_threads — кількість потоків що виконують async-задачі. Для IO-bound навантажень оптимум часто близький до кількості ядер CPU; збільшення понад це майже не дає виграшу але додає overhead на context-switching та боротьбу за work-stealing. Для CPU-bound через spawn_blocking — впливає менше бо обчислення йдуть на blocking pool.

max_blocking_threads — обмеження blocking pool куди йдуть spawn_blocking задачі. Якщо CPU-bound задач багато і обмеження мале — задачі чекатимуть у черзі, час виконання зросте. Занадто велике значення створить тиск на ОС через зайві потоки.

concurrency (кількість одночасних задач у buffer_unordered) — обмежує скільки запитів виконується паралельно. Замало — недовикористовуємо мережу і CPU; забагато — перевантажуємо мережу, отримуємо timeout, або забиваємо blocking pool. Для HTTP-bound зазвичай 8-32 оптимально.

thread_stack_size — впливає на пам'ять але не на швидкість для типових задач. Має значення лише для глибокої рекурсії або великих стек-allocated структур.

flavor (current_thread vs multi_thread) — для пакетних обробок з багатьма одночасними задачами multi_thread виграє у разів. Для одиночного запиту різниці майже немає, current_thread навіть виграє за рахунок відсутності синхронізації між workers.

2. Чому велика кількість потоків не завжди сприяє швидкому асинхронному виконанню?

Кожен додатковий потік — це накладні витрати: пам'ять під стек, контекст у планувальнику ОС, кеш-промахи при context switching. Для IO-bound задач один потік може обслуговувати тисячі асинхронних з'єднань через event loop, тому додавання потоків мало що дає. Для CPU-bound кількість корисних потоків обмежена кількістю фізичних ядер — більше потоків лише призведе до контеншена і втрати продуктивності через постійне перемикання. Також збільшується ймовірність помилок: гонки між потоками за спільні ресурси, contention на синхронізаційних примітивах (Mutex, atomic), false sharing у кеш-лініях. Закон Амдала також обмежує користь: послідовна частина роботи не масштабується.

3. Чому в tokio існує blocking thread pool?

Worker threads в tokio оптимізовані для асинхронних задач і їх кількість невелика (зазвичай за кількістю ядер). Якщо у такому потоці запустити блокувальну операцію (синхронний syscall, важке обчислення, sleep), він застрягне і не зможе обслуговувати інші асинхронні задачі. При невеликій кількості workers це швидко "повісить" увесь рантайм.

Blocking pool — окремий пул потоків що зростає за потребою (за замовчуванням до 512). У нього виносять усе що блокує потік. tokio::task::spawn_blocking повертає JoinHandle який можна await — таким чином блокувальна робота йде на blocking pool, а worker-потоки залишаються вільними для асинхронних задач. Це і є фундаментальне розділення обов'язків між двома пулами в tokio.
