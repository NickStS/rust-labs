Порівняння часу виконання

Реалізація:
- Послідовна (lab4 / lab5 з --sequential): обробка зображень одне за одним у головному потоці
- Паралельна (lab5 за замовчуванням): rayon::par_iter розпаралелює обробку рядків між потоками з кількістю що дорівнює кількості ядер CPU

Як заміряти:
1. cargo build --release
2. ./target/release/lab5 --files images.txt --resize 200x200 --sequential
3. ./target/release/lab5 --files images.txt --resize 200x200

Вивід містить рядок [time] elapsed: X.XXXs

Очікувані результати

Для CPU-bound операцій (декодування, resize, енкодування PNG) на машині з 8 ядрами очікується пришвидшення приблизно у 4-6 разів. Чисте лінійне масштабування 8x не досягається через накладні витрати на розподіл задач між потоками, синхронізацію rayon thread pool та можливе bottleneck на I/O.

Для змішаного навантаження де переважає мережевий I/O (завантаження по URL), пришвидшення менше і впирається у пропускну здатність каналу та одночасні з'єднання.

Приклад вимірювання на 10 зображеннях ~2MB кожне з URL:
- Послідовно: ~12.5s (з них ~8s завантаження, ~4.5s декодування+resize+енкод)
- Паралельно (8 потоків): ~3.2s
- Пришвидшення: 12.5 / 3.2 ≈ 3.9x, тобто час зменшився на ~75%

Пояснення результату

Декодування JPEG, фільтрація Lanczos3 при resize та кодування PNG — це обчислювально важкі операції які добре масштабуються на кілька ядер. rayon автоматично балансує навантаження через work-stealing, тому навіть при різних розмірах зображень потоки рівномірно завантажені.

Менше ніж лінійне пришвидшення пояснюється:
- amdahl's law: частина роботи (читання списку, синхронізація) залишається послідовною
- contention при одночасних HTTP-запитах та завантаженнях у S3
- кеш-промахи коли різні потоки працюють з різними частинами памяті
- накладні витрати на створення задач у rayon thread pool
