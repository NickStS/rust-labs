Що було покращено у порівнянні з lab8

1. Заміна buffer_unordered на JoinSet з Semaphore
Було: stream::iter(...).buffer_unordered(N) — задачі polled на одному виконавцеві, не можуть паралельно виконуватися на різних worker-потоках.
Стало: tokio::task::JoinSet + Semaphore::acquire_owned — кожна задача спавниться через JoinSet::spawn, тому tokio може розкласти їх по worker-потоках і виконувати справді паралельно. Семафор обмежує кількість одночасних задач щоб не перевантажити blocking pool і мережу.

2. Стримінг тіла HTTP-відповіді
Було: resp.bytes() — завантажує всю відповідь у Vec<u8> одним блоком.
Стало: resp.bytes_stream() з futures::TryStreamExt + накопичення у попередньо алокований Vec — дозволяє почати запис до файлу або обробку до повного прийому. Для нашого пайплайну ми все одно мусимо мати весь файл у памʼяті бо image::decode не вміє стрімити, але стримінг знижує піковий обʼєм виділеної памʼяті у HTTP-стеку.

3. Включена error_for_status() при HTTP-запиті
Було: status просто логувався у вивід.
Стало: resp.error_for_status() — будь-який 4xx/5xx стає AppError::Http а не безшумно записується "помилкова" відповідь у файл.

4. Pool reuse у reqwest::Client
Додано pool_max_idle_per_host = concurrency щоб переиспользувати TCP-зʼєднання з одним хостом між паралельними завантаженнями.

5. Async читання вхідного файлу через BufReader
Локальні файли тепер відкриваються через tokio::fs::File з BufReader замість блокувального std::fs::read.

Чому НЕ використано

mio — це низькорівнева event loop бібліотека. tokio вже використовує mio під капотом для async-IO. Прямо взаємодіяти з mio потрібно лише при написанні власного рантайму або реалізації специфічного протоколу з нестандартним event-driven паттерном. Для нашого CLI обробника зображень це надлишково.

futures::stream::for_each_concurrent — могла б замінити JoinSet, але for_each_concurrent виконує всі задачі у одному "host" task на одному worker-потоці. JoinSet::spawn створює окремі tokio-task, які можуть бути розподілені виконавцем по різним worker-потокам. Для нашого випадку де у задачі є spawn_blocking це не критично, але JoinSet дає більше гнучкості і явного контролю над JoinError.

tokio::sync::Mutex — спільний мутабельний стан між задачами відсутній. Семафор єдиний примітив синхронізації що тут потрібен.

tokio::select! — не використано бо немає сценарію очікування першої з кількох подій. Є один лінійний пайплайн на задачу: fetch -> cpu -> upload.
