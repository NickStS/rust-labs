1. Що таке регресійне тестування? Для чого воно використовується?

Регресійне тестування — це повторне виконання раніше пройдених тестів після внесення змін у код. Його мета — переконатися що нова функціональність або виправлення багу не зламали існуючу поведінку. Воно використовується щоб виявити регресії: ситуації коли раніше працюючий функціонал перестав працювати після змін. Зазвичай регресійні тести автоматизуються і запускаються у CI на кожному коміті або PR.

2. Поясніть принципову різницю між технічною серйозністю дефекту (Severity) та його бізнес-пріоритетом (Priority).

Severity описує наскільки сильно баг впливає на роботу системи з технічної точки зору: чи призводить до краху, втрати даних, неправильних обчислень тощо. Це обєктивна характеристика яку визначає переважно інженер. Priority описує наскільки терміново баг треба виправити з точки зору бізнесу: чи блокує він важливих клієнтів, чи зриває реліз, чи впливає на репутацію продукту. Її визначає продукт-менеджер. Баг може мати високу Severity але низький Priority — наприклад крах функції якою ніхто не користується. Або навпаки — низька Severity але високий Priority, наприклад опечатка у назві компанії на головній сторінці.

3. Чому при виправленні багів, зафіксованих у трекері, важливо дотримуватися принципу атомарності коммітів?

Атомарний коміт — це коміт що містить одне логічно завершене виправлення одного бага. Це важливо тому що: дозволяє легко зробити revert конкретного фіксу якщо він викликав регресію без втрати інших змін; спрощує code review бо рев'юер бачить лише релевантні зміни; дозволяє точно прив'язати коміт до тікета у трекері і легко знайти його у git історії; полегшує bisect для пошуку причини нової регресії; робить історію проекту читабельнішою для майбутніх розробників.
