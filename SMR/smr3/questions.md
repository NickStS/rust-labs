1. Дослідити можливі проблеми багатопотокового виконання, такі як гонки ресурсів (не тільки даних) та взаємного блокування, методи діагностики та усунення цих проблем.

Гонки даних виникають коли два потоки одночасно звертаються до одних даних і хоча б один пише без синхронізації — Rust забороняє їх на рівні системи типів через правила запозичення та трейти Send/Sync. Гонки ресурсів є ширшим поняттям і можливі навіть при коректній синхронізації окремих доступів. Класичний приклад — check-then-act коли потік перевіряє значення лічильника і потім модифікує, а між цими двома операціями інший потік встигає змінити стан. Кожен окремий доступ під мютексом коректний але загальна логіка некоректна. Лікується атомарними read-modify-write операціями (compare_exchange, fetch_add) або тримання блокування протягом всієї послідовності операцій. Дедлок виникає коли два і більше потоків чекають один на одного по колу — потік A тримає мютекс M1 і чекає M2, потік B тримає M2 і чекає M1. Уникається через узгоджений порядок взяття блокувань, використання try_lock з таймаутом, або об'єднання дрібних мютексів у один. Livelock — потоки активно реагують один на одного але не прогресують, схоже на двох людей що ввічливо звільняють дорогу один одному. Голодування виникає коли потік не отримує доступ до ресурсу через постійну конкуренцію — наприклад письменник у RwLock не може взяти блокування поки є читачі. Для діагностики використовують ThreadSanitizer (через RUSTFLAGS=-Zsanitizer=thread на nightly), Loom для систематичного дослідження всіх можливих interleavings у тестах, parking_lot::deadlock_detector який відстежує цикли у графі очікування, а також Miri що виявляє data races у однопотокових тестах async-коду.

2. Дослідити можливості неблокуючих структур даних у Rust, випадки їх використаття та відмінності між їх імплементаціями.

Неблокуючі структури — це такі що не використовують мютекси для синхронізації, замість цього застосовують атомарні операції процесора. У стандартній бібліотеці є атомарні скаляри: AtomicBool, AtomicUsize, AtomicI32 тощо з операціями load, store, fetch_add, fetch_sub, compare_exchange. Вони підходять для лічильників, флагів, простих lock-free алгоритмів. Крейт crossbeam надає більш складні структури: ArrayQueue для bounded MPMC чергі на базі ring buffer з атомарними головою та хвостом, SegQueue для unbounded черги з сегментованим списком, AtomicCell<T> для lock-free атомарного зберігання довільного типу що поміщається у машинне слово, ShardedLock як заміна RwLock з кращою масштабованістю на читання. Крейт arc-swap надає ArcSwap для атомарного оновлення спільних незмінних даних — корисно для конфігурацій що рідко змінюються але часто читаються. dashmap дає concurrent HashMap зі sharding підходом — розбиває таблицю на сегменти кожен зі своїм блокуванням, що зменшує contention порівняно з Arc<Mutex<HashMap>>. Існують також lock-free черги типу crossbeam-deque (work-stealing для виконавців) та flume (channel з atomic-based notifier).

3. Описати в чому переваги неблокуючих струтур даних, яким чином досягається ефект неблокування.

Головна перевага — масштабованість при високій конкуренції. Звичайний Mutex при одночасному доступі багатьох потоків призводить до контеншена: потоки чекають на ядрі ОС, виконують context switch, втрачають кеш, що різко знижує продуктивність. Неблокуючі структури використовують атомарні CPU-інструкції типу CMPXCHG (на x86) або LL/SC (на ARM) які виконуються за один такт без виходу у kernel. Якщо операція успішна — потік просто продовжує. Якщо інший потік вже встиг змінити значення — операція повторюється у циклі (це називають CAS-loop або lock-free retry). Завдяки відсутності блокувань уникається проблема inversion of priority (коли низькопріоритетний потік тримає блокування потрібне високопріоритетному), не може виникнути дедлок, потоки не блокуються при паніці іншого потоку. Існує два рівні гарантій: lock-free означає що принаймні один потік завжди прогресує, wait-free означає що кожен потік завершує операцію за обмежений час незалежно від інших. Ці гарантії досягаються через ретельну роботу з memory ordering та коректне використання fence-операцій. Недолік — складність реалізації, ABA-проблема (значення змінилося і повернулося до попереднього між двома читаннями), необхідність у hazard pointers або epoch-based reclamation для безпечного звільнення памʼяті.

4. Дослідити та порівняти популярні бібліотеки в екосистемі Rust для роботи із каналами. Розглянути варіанти їх принцип роботи, випадки використання.

std::sync::mpsc — стандартна реалізація multi-producer single-consumer каналу. Sender можна клонувати для багатьох виробників, Receiver — лише один. Підтримує bounded (sync_channel) та unbounded варіанти. Простий та достатній для базових сценаріїв але не дуже швидкий, не підтримує select між кількома каналами, не має MPMC варіанту.

crossbeam-channel — швидша заміна стандартного mpsc. Підтримує MPMC (multi-producer multi-consumer): можна клонувати і Sender і Receiver. Має макрос select! для очікування першої події з кількох каналів. Bounded канали реалізовано через ring buffer, unbounded — через сегментований список. Активно використовується у системах з високим throughput (тисячі повідомлень за секунду). Поведінка повністю синхронна — recv блокує потік, тому не підходить для async-коду.

flume — компроміс між простотою API mpsc та швидкістю crossbeam. Підтримує і sync і async API через one і той же канал. Швидкий, MPMC, з підтримкою select. Зручно мігрувати з sync-коду на async без зміни типу каналу.

tokio::sync::mpsc — асинхронний MPSC канал від tokio. send.await не блокує потік, чекає через waker. Має варіанти bounded (закінчуються send.await до звільнення місця) та unbounded. Стандартний вибір для async-коду на tokio.

tokio::sync::broadcast — кожне повідомлення отримують усі підписники. Якщо отримувач не встигає, старі повідомлення для нього губляться (lagging). Підходить для подійних архітектур.

tokio::sync::watch — single-producer multi-consumer для конфігурацій або станів які треба передавати усім споживачам. Кожен отримувач бачить тільки останнє значення.

tokio::sync::oneshot — для передачі одного значення між двома задачами. Корисно для request-response патернів.

async-channel — від async-std команди, схожий на flume, мультиплатформенний async MPMC канал. Може використовуватися з будь-яким async-рантаймом.

postage — async канали з широким API: mpsc, broadcast, oneshot, watch — мінімум залежностей, підходить коли не хочеш тягнути весь tokio.
