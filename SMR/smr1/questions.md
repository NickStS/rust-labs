1. Дослідити способи обійти системи володінь та запозичень у Rust у однопотоковому та багатопотокових контекстах. Описати як вони працюють та завдяки чому вдається не порушувати гарантії запозичень та володінь.

У однопотоковому контексті основними інструментами є Cell, RefCell, Rc та сирі вказівники. Cell зберігає Copy типи і дозволяє читати та записувати через спільне посилання — гарантії не порушуються бо назовні ніколи не повертаються посилання на внутрішні дані, тільки копії. RefCell працює з будь-якими типами і перевіряє правила запозичення у рантаймі через лічильники: якщо взяти borrow_mut коли є інші borrow, виникає паніка. Rc дає спільне володіння через підрахунок посилань але без можливості мутації — для мутації використовують Rc<RefCell<T>>. Сирі вказівники повністю обходять borrow checker але вимагають unsafe для разіменування. Усі ці типи побудовані на UnsafeCell і всередині використовують unsafe, але через безпечний публічний API забезпечують дотримання правил у рантаймі. У багатопотоковому контексті аналогом Rc є Arc з атомарним лічильником. Для мутації використовують Mutex (ексклюзивний доступ), RwLock (багато читачів або один писач), AtomicUsize/AtomicBool тощо для lock-free операцій. Усі вони повертають guard-обʼєкти через які компілятор відстежує час життя блокування. Канали (mpsc, crossbeam) натомість передають володіння між потоками — у будь-який момент значення належить лише одному потоку.

2. Дослідити методи зручного перетворення даних між різними типами. Дослідити трейти From/Into, TryFrom/TryInto, AsRef/AsMut, Borrow, Deref/DerefMut. Відмінності між наведеними трейтами.

From і Into є базовими трейтами для конвертації що завжди успішна — From::from приймає значення одного типу і повертає інший. Into реалізується автоматично для всіх типів що мають From завдяки blanket impl. Для конвертацій що можуть провалитися існують TryFrom і TryInto які повертають Result. AsRef і AsMut надають дешевий доступ до значення як до іншого типу через посилання, без жодних алокацій або клонування — використовуються коли функція хоче приймати кілька типів схожих на цільовий, наприклад параметр impl AsRef<Path> приймає і &str, і String, і PathBuf. Borrow схожий на AsRef але має додаткову семантичну гарантію: значення Borrow має поводитися еквівалентно з точки зору Eq, Ord та Hash. Це використовується у HashMap де ключі-String зберігаються але пошук map.get("key") приймає &str. Deref відрізняється тим що призначений саме для smart pointer типів — він дає автоматичне розіменування. Завдяки тому що Box, Rc, Arc, String реалізують Deref, можна писати box.method() і компілятор автоматично перетворить це на (*box).method(). DerefMut — аналог для мутабельного доступу. Поширена помилка — реалізовувати Deref для звичайних структур-агрегатів, але це призначено лише для smart pointers.

3. Дослідити крейти екосистеми Rust, які допомагають позбутися бойлерплейту при імплементації конвертації.

derive_more — найвідоміший крейт що автоматично реалізує From, Into, AsRef, AsMut, Display, Debug, Add, Mul, Deref та інші для newtype-обгорток одним атрибутом. thiserror реалізує From через атрибут #[from] усередині варіантів enum-помилки, що дозволяє автоматично конвертувати одні помилки у інші через ?-оператор. anyhow надає універсальний тип помилки що приймає будь-який Error без явних реалізацій From. serde не зовсім про конвертацію типів між собою але вирішує конвертацію між Rust-структурами та форматами як JSON, TOML, YAML через derive макроси Serialize та Deserialize. bytemuck надає safe API для побайтової конвертації між типами однакового розміру через трейти Pod і Zeroable замість небезпечного transmute. ref-cast дозволяє безпечно перетворити &Inner у &Newtype для repr(transparent) обгорток.

4. Дослідити випадки використання та обмеження макросів у Rust. Навести приклади із проектів з відкритим кодом та Rust std. Відмінності процедурних та декларативних макросів.

Декларативні макроси оголошуються через macro_rules! і працюють як патерн-матчинг на синтаксичних деревах. Зі стандартної бібліотеки приклади — vec! для створення колекцій з різною кількістю елементів, println! та format! з форматуванням, assert! та debug_assert! що додають інформацію про файл і рядок, thread_local! для оголошення TLS-змінних. Процедурні макроси є складнішими і працюють як функції від TokenStream до TokenStream, бувають трьох типів. Derive-макроси як serde::Serialize, thiserror::Error або clap::Parser генерують impl-блоки на основі структури типу. Атрибутні макроси як tokio::main, async_trait::async_trait або rocket::get модифікують код навколо елемента до якого застосовані. Function-like макроси як sqlx::query! або html! у dioxus приймають довільний синтаксис та генерують код. Декларативні макроси простіші, працюють у тому ж крейті без окремої компіляції, але обмежені у виразності — не вміють створювати нові ідентифікатори з частин або аналізувати типи. Процедурні значно потужніші бо мають повний доступ до синтаксичного дерева, але вимагають окремого крейту з proc-macro = true та збільшують час компіляції.

5. Дослідити обмеження функціоналу макросів у Rust. Описати випадки, коли використовувати макроси недоцільно.

Декларативні макроси у Rust є гігієнічними — ідентифікатори створені усередині макросу не конфліктують з ідентифікаторами місця виклику. Це підвищує безпеку але обмежує генерацію коду. Активне використання макросів збільшує час компіляції бо вони рекурсивно розгортаються; процедурні макроси гірші за цим показником бо є окремими крейтами що компілюються до основного коду. Помилки у згенерованому коді часто показуються на рядку макровиклику замість справжнього місця проблеми, що ускладнює відлагодження — частково допомагає cargo expand. IDE-підтримка для складних процедурних макросів буває обмеженою, rust-analyzer не завжди коректно розуміє згенерований код. Макроси недоцільно використовувати коли можна обійтися дженериками з трейтами бо тип-система дає кращі повідомлення про помилки і автодоповнення. Не варто використовувати їх для простих повторюваних шаблонів які можна виразити функціями. Макроси не підходять коли логіка залежить від рантайм-значень бо вони розгортаються тільки на компіляції. У публічному API краще уникати без необхідності щоб клієнтський код залишався зрозумілим. Не варто писати макрос для речей які покриваються стандартними абстракціями — наприклад замість макросу для конвертації краще реалізувати From або Into.
