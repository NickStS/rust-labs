1. Дослідити можливі різних імплементацій асинхронних Rust рантаймів, окрім tokio. Зробити порівняльну характеристику, описати переваги та недоліки кожного із них.

async-std — рантайм створений як асинхронна заміна стандартної бібліотеки. API максимально схоже на std: async_std::fs::File, async_std::net::TcpStream тощо мають ті самі сигнатури що і синхронні аналоги. Це знижує бар'єр входу для розробників звиклих до std. Підтримує task::spawn для багатопотокового виконання, channels, синхронізаційні примітиви. Недоліки: значно менша екосистема ніж у tokio, повільніший розвиток, частина крейтів несумісна або потребує адаптерів. Більшість бібліотек у Rust пишуться під tokio, тому використання async-std часто змушує писати власні обгортки.

smol — мінімалістичний рантайм створений тим самим автором що і async-std. Складається з невеликих окремих крейтів: async-task для управління задачами, async-io для async-IO, async-channel для каналів, blocking для блокувальних операцій, polling як обгортка над epoll/kqueue/IOCP. Одна з ключових ідей — модульність: можна взяти лише те що потрібно. Дуже легкий і швидкий старт, добре підходить для embedded або CLI-утиліт. Недоліки: екосистема ще менша, потребує більше ручної конфігурації для складних застосунків.

monoio — асинхронний рантайм від ByteDance заснований на io_uring (Linux) і IOCP (Windows). Використовує модель thread-per-core: кожен потік має власну event loop без work-stealing. Це дає кращу cache locality і знижує contention але вимагає інакшого підходу до архітектури. Дуже високопродуктивний для serverless та network-intensive задач. Недоліки: потребує сучасного ядра Linux для io_uring, futures не Send бо thread-per-core, не сумісний з більшістю tokio-екосистеми.

glommio — інший thread-per-core рантайм від Datadog заснований виключно на io_uring. Орієнтований на storage та database навантаження де I/O швидкість критична. Має інтеграцію з direct I/O оминаючи page cache. Недоліки: тільки Linux, тільки io_uring, дуже специфічний фокус.

embassy — async рантайм для embedded/no_std середовищ. Працює без аллокатора, підтримує мікроконтролери ARM Cortex-M, RISC-V, ESP32. Доводить що async підходить навіть для систем з 16KB RAM. Недоліки: інша модель програмування, обмежений набір підтримуваних платформ.

actix-rt — рантайм-обгортка над tokio оптимізована під actix-web. По суті це той самий tokio але з налаштуваннями для веб-фреймворку. Майже не використовується окремо.

futures::executor::block_on — не справжній рантайм, а простий виконавець з futures крейту. Підходить для виконання одиничного future у синхронному коді або у тестах. Не підтримує асинхронне IO без додаткового реактору.

2. Дослідити способи відміни асинхронної задачі в Rust і чому це є складним завданням у Rust екосистемі.

В Rust асинхронна задача скасовується буквально через її drop. Якщо JoinHandle або сама future буде відкинута, рантайм просто перестає її polling. Цим Rust відрізняється від інших мов де скасування потребує окремого механізму. Здавалося б це просто, але насправді породжує багато проблем. Drop може статися у будь-якій точці await — перед, посередині або після. Усі ресурси що утримує future повинні коректно звільнятися навіть якщо вона завершилася на половині операції. Якщо future зробила HTTP-запит і чекає відповідь, її drop викине з'єднання у непередбачуваному стані. Якщо вона тримала Mutex guard — guard звільниться, але стан що захищав може бути не оновлений до консистентного. Якщо була у середині транзакції — транзакція не буде ні закомічена ні відкатена явно. Це називають "cancellation safety" — функція є cancel-safe тільки якщо її скасування у будь-який момент залишає програму у валідному стані. Більшість простих async функцій є cancel-safe тривіально. Складні комбінатори типу tokio::select! явно вимагають cancel-safe гілок бо поза переможцем інші відразу скасовуються. Для контрольованого скасування існує tokio_util::sync::CancellationToken: задачі періодично перевіряють token.is_cancelled() або await на token.cancelled() через select. Це дає graceful shutdown — задача знає що її просять зупинитися і може коректно завершити критичну роботу перед виходом. Складність також у тому що Rust не має винятків, тому скасовану задачу не можна "просигналізувати" виключенням як це роблять у Java або Python — все відбувається через drop і CancellationToken стає узгодженим механізмом. Існує проблема з блокувальними операціями винесеними у spawn_blocking — їх не можна скасувати взагалі, бо вони виконуються на справжніх потоках ОС.

3. Дослідити модель акторів для конкурентного виконання задач. Порівняти із звичайним асинхронним виконанням. Знайти спільні та відмінні риси.

Модель акторів — це парадигма де одиницею паралелізму є актор: ізольований обʼєкт що має приватний стан і спілкується з іншими акторами тільки через асинхронні повідомлення. Кожен актор обробляє повідомлення послідовно одне за одним, тому всередині актора немає гонок даних і не потрібні мютекси. Між акторами немає спільної памʼяті, тільки канали повідомлень. Концепція походить з Erlang та активно використовується в Akka (Scala) і Orleans (.NET). У Rust моделі акторів представлені крейтами actix, ractor, kameo, tokio-actor. Найвідоміший actix дозволяє визначати актори через структури з реалізацією трейту Actor, повідомлення через Message, обробники через Handler<M>. Актори мають власну адресу через яку інші можуть надсилати повідомлення. Спільне з звичайним async — обидва підходи орієнтовані на конкурентність без спільної памʼяті, обидва дають вільність від data race на рівні системи типів, обидва потребують async рантайму. Ключові відмінності: у звичайному async задачі можуть передавати дані через спільні структури під Mutex чи Arc, актори ж тримають увесь свій стан приватним і доступ до нього лише через повідомлення. Async задачі є тимчасовими (виконав і завершив), актори зазвичай довго живуть і обробляють потік повідомлень. Async код використовує await для очікування, актори обробляють повідомлення з черги. Async більш гнучкий і ідіоматичний для Rust бо інтегрується з мовними засобами безпосередньо. Актори додають структуру для систем з багатьма незалежними станами що потребують стійкої ізоляції — наприклад симуляції, ігри, комплексні стейтфул протоколи. Недолік моделі акторів у Rust полягає в тому що вона відчувається як накладений зверху патерн а не як природна частина мови, бо Rust не має вбудованої підтримки на відміну від Erlang. Async з каналами часто є простішою і чистішою альтернативою: фактично актор — це просто задача що крутиться у циклі і обробляє повідомлення з mpsc-каналу.
